# Анализ проекта skazka-v3

## Кратко
- Frontend: React + Vite, один серверный `server.ts` как API + Vite middleware.
- Backend хранит истории в SQLite (`stories.chapters_json`) и обслуживает AI‑генерации через OpenRouter (текст) и N1N (картинки).
- Клиент генерирует мир/героев/главы и сохраняет историю через `/api/stories`, а ассеты грузит на `/api/assets/upload`.

## Потоки
1. Пользователь выбирает жанр/возраст → `DoorSelect` → `StoryWizard`.
2. `StoryWizard` вызывает `/api/ai/generate-world` → получает 3 мира → генерирует обложки → выбирает мир → генерирует героя/антагониста и их изображения.
3. Создание первой главы: `App.tsx` вызывает `generateChapter` и `generateImage`, сохраняет в `/api/stories`.
4. Чтение: `StoryReader` показывает главу и 3 выбора, при выборе генерирует следующую главу и добавляет в `chapters`.
5. Дерево глав строится в `StoryContents` по `parentChapterId`/`choiceIndex`.

## Проблемы и риски (по приоритету)

1. **Неверный URL для аудио‑ассетов**
   В `server.ts` аудио сохраняется в `data/audio`, а URL формируется как `/assets/audios/...`, который мапится на `data/audios` (не существует). Аудио не загрузится.
   Файл: `server.ts` (upload handler).

2. **Смешение типов времени приводит к неправильной сортировке**
   Клиент отправляет `createdAt/updatedAt` как ISO‑строки, сервер хранит как INTEGER. В БД получается смесь чисел и строк, а `ORDER BY updated_at` может давать неправильный порядок.
   Файлы: `server.ts`, `src/App.tsx`, `src/types.ts`.

3. **Кэш‑заголовок `immutable` ломает обновление ассетов**
   Статик `/assets` отдаётся с `immutable`. Ассеты имеют стабильные имена (`world_${id}`, `hero_${id}`, `scene_${chapter.id}`), и при перегенерации картинки браузер может держать старый файл до 7 дней.
   Файл: `server.ts` (static assets).

4. **Дублирующая логика промпта и неиспользуемая ветка генерации**
   Серверный `/api/ai/generate-chapter` содержит огромный промпт (3000–3500 символов), но клиент всегда вызывает ветку `prompt + systemPrompt`, которая просит 800–1200 символов. Большая серверная ветка фактически не используется.
   Файлы: `server.ts`, `src/services/ai.ts`.

5. **Таблица `chapters` не используется**
   Есть схема `chapters`, но записи туда не пишутся. Удаление идёт по `stories.chapters_json`. Таблица мёртвая, это лишняя сложность.
   Файл: `server.ts`.

## Логические несоответствия/мелкие риски

- `AI_PROVIDER` вычисляется, но текст всегда через OpenRouter, картинки всегда через N1N. Реальный роутинг не соответствует переменной.
  Файл: `server.ts`.

- `RequestQueue` ждёт минимум 1 секунду **до** и **после** каждого запроса, что делает конвейер заметно медленнее, чем нужно.
  Файл: `src/services/ai.ts`.

- `StoryContents` строит дерево с вложенным `find` в цикле, что даёт O(n²). На больших историях может тормозить.
  Файл: `src/components/StoryContents.tsx`.

- В `MyStories` параметры `onDeleteStory` и `onNewStory` переименованы с `_`, но фактически удаление историй не реализовано в UI, хотя API есть.
  Файл: `src/components/MyStories.tsx`.

3. Убрал `immutable` или внедрил cache‑busting для ассетов.
4. Упростил `/api/ai/generate-chapter`, оставив одну актуальную ветку генерации.
5. Решил судьбу таблицы `chapters`.
